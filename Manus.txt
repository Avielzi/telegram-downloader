1. קלט ובדיקות

   * חלץ את ה־ZIP לתיקיית עבודה נקייה.
   * זהה את root האמיתי של הפרויקט (אם יש שכבת תיקייה אחת מעל, תקן כך שהתוכן יהיה ברמת השורש הנכונה).

2. ניקוי והחלפה מלאה (No leftovers)

   * בצע checkout ל־branch הראשי של הריפו.
   * נקה את עץ העבודה לחלוטין כך שלא יישארו קבצים ישנים:

     * מחק כל קובץ/תיקייה שאינם חלק מהגרסה החדשה, כולל קבצים שנמחקו בגרסה החדשה.
     * אל תמחק: .git, ואל תשבור הגדרות CI אם הן קיימות בגרסה החדשה.
   * העתק את תוכן הגרסה החדשה לשורש הריפו.
   * ודא שהתוצאה הסופית משקפת 1:1 את תוכן ה־ZIP (אחרי תיקון root אם נדרש).

3. README ומטא-דאטה

   * עדכן/החלף README כך שישקף את הגרסה החדשה:

     * שם הפרויקט, תיאור קצר, דרישות מערכת, התקנה, שימוש בסיסי, דוגמאות, Troubleshooting.
     * אם קיימים קבצי docs/ או wiki references — התאמה בהתאם.
   * עדכן קבצי גרסה אם קיימים (לפי מה שיש בפרויקט): package.json / pyproject.toml / setup.cfg / go.mod / Cargo.toml וכו’.
   * עדכן/צור CHANGELOG.md (אם מתאים) עם כניסה לגרסה החדשה.

4. טיפול בגרסאות קודמות (אופציונלי, לא בקוד הראשי)

   * אם קיימת היסטוריה של גרסאות בתוכן הישן:

     * אל תשאיר קבצים ישנים ב־main.
     * שמור אותם באחת מהאפשרויות:
       A) Tag + GitHub Release (מומלץ): צור tag לגרסה הקודמת האחרונה, ו־release notes קצרות.
       B) Branch archive/<version>: העבר את מצב הקוד הישן ל־branch ארכיון.
       C) תיקיית /archive בתוך הריפו *רק אם המשתמש ביקש* — ברירת מחדל: לא.
   * בחר A כברירת מחדל אם אפשר.

5. אימות (Hard checks)

   * הרץ diff/השוואה: רשימת קבצים סופית בריפו = רשימת קבצים מהגרסה החדשה (למעט .git ו־GitHub config כמו .github אם קיים בחדשה).
   * חפש “orphan files”: קבצים שלא קיימים ב־ZIP אך קיימים בריפו — חייב להיות 0.
   * ודא שאין כפילויות גרסאות (למשל telegram-downloader-v2.1 לצד v2.2) בתוך main.

6. Commit + Push

   * Commit אחד נקי עם הודעה בפורמט:
     "chore(release): update to vX.Y.Z from zip (clean replace)"
   * Push ל־origin על branch הראשי.
   * אם יצרת tag/release — דחוף גם אותם.
